Why This Matters

The Problem with Embedding API Keys

If you've ever built an AI-powered e-learning activity by pasting an API key directly into your HTML or JavaScript file, you've accidentally left the front door unlocked.

What is an API Key?

Think of an API key like a credit card number for a service. When you use a paid AI service like Grok or ChatGPT, you're given a key that authorizes you to use it — and gets billed for every request.

If someone finds your key, they can use the AI service on your bill — and you'd have no idea until you get a surprise invoice.

🚨 Never paste an API key directly into HTML, JavaScript, or any file that gets published to the web. It can be found by anyone who opens DevTools (right-click → Inspect) in their browser.
The Solution: A Middleman Server

The fix is to put your API key on a private server that you control. Your course or learning activity talks to your server, and your server talks to the AI. The key never reaches the learner's browser.

Learner's Browser
Your Course / App
sends message
Your Server
API Key lives here 🔒
forwards to AI
AI Service
Grok / ChatGPT

The learner only ever sees your server's response — never the key itself. It works exactly like ordering food at a restaurant. You (the learner) tell the waiter (your server) what you want. The waiter calls the kitchen (the AI). You get your food. You never needed the kitchen's private phone number.

What You'll Build

In this guide you'll set up a complete secure pipeline: a backend server hosted for free on Render, a frontend HTML file hosted on GitHub Pages, and a locked-down connection between the two.

Cost: Everything in this guide is free. Render's free tier, GitHub Pages, and a Grok API key are all you need.
Phase 1

Accounts & Tools

Before writing a single line of code, get these four things in place. This is a one-time setup.

1. Install Node.js

Node.js is the engine that runs your backend server on your computer during development. Think of it as the runtime environment — like how you need a PDF reader to open PDFs, you need Node to run server-side JavaScript.

  1. Go to nodejs.org and download the LTS version (the left button).
  2. Run the installer with all default settings. Leave the "Native Tools" checkbox unchecked.
  3. Open a new PowerShell window and run node -v then npm -v — both should show version numbers.
PowerShell tip: If npm gives a security error, run this once:
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
2. Create a GitHub Account

GitHub is where your code lives online. It also powers GitHub Pages — the free hosting for your frontend. If you already have an account, you're good.

Your GitHub username becomes part of your public URL: https://yourusername.github.io

3. Create a Render Account

Render is where your backend server will live in the cloud — running 24/7 at a real URL. Sign up at render.com using your GitHub account so they're linked automatically.

Free tier note: Render's free servers "spin down" after 15 minutes of inactivity. The first request after a sleep takes about 50 seconds. This is fine for prototypes and learning tools.
4. Get Your AI API Key

For this guide we used Grok (xAI). Get your key at console.x.ai. The same concept applies to any AI provider — OpenAI, Anthropic, Google, etc.

Treat this like a password. Never share it in chat, email, or any file that gets published. If it gets exposed, regenerate it immediately at your provider's console.
Phase 2

Build the Backend

The backend is a tiny server with one job: receive messages from your course, forward them to the AI using your hidden key, and return the response.

Create Your Project Folder

In PowerShell, create a clean folder structure:

cd C:\Users\YourName
mkdir id-simulation
cd id-simulation
mkdir backend
cd backend
npm init -y

The last command creates a package.json file — think of it as the ID card for your project.

Install Dependencies

Run this one command:

npm install express cors dotenv
PackageWhat it does
expressCreates the web server and defines your endpoint
corsControls which websites are allowed to talk to your server
dotenvReads your secret API key from the hidden .env file
Create Your .env File

Create a file named exactly .env in your backend folder. This is where your key lives locally:

GROK_API_KEY=your_actual_api_key_here
PORT=3000
This file must NEVER be pushed to GitHub. Always add .env to your .gitignore before running git add .
Create server.js

This is the actual backend file. It creates one endpoint — /chat — that your frontend will call:

// Load environment variables from .env
require('dotenv').config();
const express = require('express');
const cors = require('cors');

const app = express();

// Only allow requests from your frontend URL
app.use(cors({ origin: 'https://yourusername.github.io' }));
app.use(express.json());

// The /chat endpoint — this is what your frontend calls
app.post('/chat', async (req, res) => {
  const { messages } = req.body;
  const response = await fetch('https://api.x.ai/v1/chat/completions', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      // The key is read from .env — never hardcoded here
      'Authorization': `Bearer ${process.env.GROK_API_KEY}`
    },
    body: JSON.stringify({ model: 'grok-3-mini', messages })
  });
  const data = await response.json();
  res.json(data);
});

app.listen(process.env.PORT || 3000, () =>
  console.log('Server running'));
Test Locally

Start your server:

node server.js

You should see Server running in the terminal. In a second PowerShell window, test it:

Invoke-RestMethod -Uri "http://localhost:3000/chat" -Method POST -ContentType "application/json" -Body '{"messages":[{"role":"user","content":"say hello"}]}'

If you get a response back from the AI, your backend works.

✓ What just happened: Your PowerShell acted as a "fake browser" and called your local server. The server used your hidden API key to talk to Grok and returned the response. Your key never left your machine.
Phase 3

Build the Frontend

The frontend is the part learners see and interact with — in our case, a chat interface. It's a single HTML file with no server required to run it.

Create the Frontend Folder

Inside your id-simulation folder (not inside backend), create a frontend folder and place your index.html there:

id-simulation/
├── backend/
│   ├── .env          ← secret key (never on GitHub)
│   ├── .gitignore
│   ├── package.json
│   └── server.js
└── frontend/
    └── index.html    ← your course/activity UI
The Key Line in Your Frontend

Your index.html communicates with your backend through one line. During development it points to your local server:

// During local development:
const API_URL = 'http://localhost:3000/chat';

After deploying, you update this to your live Render URL:

// After deploying to Render:
const API_URL = 'https://your-app-name.onrender.com/chat';

That one line is the entire connection between frontend and backend. Everything else in your HTML file can be pure creative work.

How the Frontend Sends a Message

When a learner types a message and hits send, your JavaScript does this:

const response = await fetch(API_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ messages: conversationHistory })
});

Notice: no API key anywhere in this code. The frontend only knows your server's URL — nothing about Grok or your credentials.

Phase 4

Push to GitHub

Before deploying to the cloud, your code needs to be on GitHub. This is also where the most common mistake happens — accidentally uploading your secret key.

⚠️ The Golden Rule: .gitignore First

Always — always — create your .gitignore file before running git add .. If you add files first, Git stages everything including your secrets.

Your .gitignore file should contain exactly these two lines:

node_modules/
.env
The right order matters:
1. git init
2. Create .gitignore with the exclusions
3. git add .
4. git commit
Push the Backend
  1. Create a new empty repo on GitHub (no README, no license).
  2. In your backend folder, initialize Git and commit your clean files.
  3. Connect to GitHub and push.
git init
git add .
git status   # verify — should only show .gitignore, package.json, server.js
git commit -m "initial backend commit"
git remote add origin https://github.com/yourusername/your-repo.git
git branch -M main
git push -u origin main
✓ Verify on GitHub: Open your repo and confirm you only see .gitignore, package-lock.json, package.json, and server.js. If .env or node_modules/ appear, something went wrong.
What if .env Gets Committed by Accident?

GitHub will block your push and warn you. If this happens:

  1. Immediately regenerate your API key at your provider's console. The old one is compromised.
  2. Remove Git's history: Remove-Item -Recurse -Force .git
  3. Manually unstage the problem files: git rm -r --cached node_modules and git rm --cached .env
  4. Re-initialize, re-add, and push cleanly.
Phase 5

Deploy to Render

Render takes your GitHub repo and turns it into a live, running server in the cloud. This is where your API key lives online — stored securely as an environment variable, not in any file.

Create a Web Service
  1. Log in to render.com and click New → Web Service.
  2. Connect your GitHub repo (id-simulation-backend).
  3. Fill in the settings below and click Deploy Web Service.

Add the Environment Variable

This is the online version of your .env file. Scroll to the Environment Variables section before clicking Deploy:

Render stores this securely on their servers. It never appears in your code or your GitHub repo. Your server.js reads it the same way it reads your local .env — via process.env.GROK_API_KEY.
Confirm It's Live

Watch the deploy logs. When you see Server running, your backend is live at a URL like:

https://id-simulation-backend.onrender.com

Your /chat endpoint is now reachable at:

https://id-simulation-backend.onrender.com/chat

Update the API_URL constant in your index.html to this URL.

Phase 6

Deploy Frontend to GitHub Pages

GitHub Pages hosts your HTML file for free at a public URL. No server needed — it serves the file directly to anyone who visits the link.

Create a Frontend Repo and Push
  1. Create a new empty repo on GitHub named id-simulation.
  2. In your frontend folder, initialize Git and push index.html.
cd C:\Users\YourName\id-simulation\frontend
git init
git add .
git commit -m "initial frontend commit"
git remote add origin https://github.com/yourusername/id-simulation.git
git branch -M main
git push -u origin main
Enable GitHub Pages
  1. Go to your repo on GitHub.
  2. Click Settings → Pages in the left sidebar.
  3. Under Source, select Deploy from a branch.
  4. Set Branch to main, folder to / (root).
  5. Click Save. Wait 1–2 minutes.

Your site will be live at:

https://yourusername.github.io/id-simulation
✓ You're done! Open that URL in any browser — the simulation is live, the backend is secure, and your API key is hidden.
Updating Your Files Later

Whenever you make changes to index.html or server.js, push the updated file to GitHub. Render and GitHub Pages both auto-deploy on every push to main.

# To update the frontend:
cd frontend
git add index.html
git commit -m "update simulation UI"
git push

# To update the backend:
cd backend
git add server.js
git commit -m "update CORS or endpoint logic"
git push

CORS Explained

CORS — Cross-Origin Resource Sharing — is a browser security rule that controls which websites are allowed to talk to your server. It's the lock on the door.

What is an "Origin"?

An origin is a combination of a protocol, domain, and port. These are all different origins:

https://kevlovesdev.github.io       ← your GitHub Pages site
https://example.com                 ← different domain
http://localhost:3000               ← local development
https://rise.com                    ← an LMS

When your browser-based course tries to call your backend, the browser checks: "Is this origin allowed?" If not, the request is blocked before it even leaves the browser.

How to Set CORS in Your Server

Open (anyone can call your server — not recommended for production):

app.use(cors()); // No restrictions

Single origin (recommended — one specific URL only):

app.use(cors({ origin: 'https://kevlovesdev.github.io' }));

Multiple origins (if you embed in multiple places):

app.use(cors({
  origin: [
    'https://kevlovesdev.github.io',
    'https://app.rise.com',
    'https://yourschool.instructure.com'
  ]
}));
How Broad Can a CORS URL Be?

This is an important question. Here's the short answer: CORS matches the origin only — not paths, subfolders, or parameters.

What you put in CORSWhat it allowsGood idea?
https://rise.com Any page on rise.com — including other people's courses ❌ Too broad
https://app.rise.com Any page on the Rise app subdomain ⚠️ Use with caution
https://kevlovesdev.github.io Only pages hosted on your personal GitHub Pages ✅ Recommended
https://yourschool.instructure.com Your specific Canvas instance only ✅ Fine for org use
Important: CORS is a browser-level protection only. It stops browsers from making unauthorized requests, but a developer with technical tools can still bypass it. CORS is about reducing accidental misuse — it's one layer of security, not a complete wall. Always pair it with rate limiting and key rotation for anything in production.

You cannot restrict to a specific subfolder path. https://rise.com/courses/my-specific-course is NOT valid as a CORS origin — the path portion is ignored. If you're hosting inside an LMS, the best approach is to host your frontend on your own GitHub Pages and embed it via an iframe.

Talking to Software Engineers

If you're working with a dev team instead of building this yourself, here's how to communicate what you need clearly and professionally.

The Vocabulary You Now Know
TermPlain English
API endpointA specific URL your app sends messages to. Like a dedicated phone line for one purpose.
Environment variableA secret stored on the server, not in the code. Like a password written on a sticky note that only the server can see.
CORSA rule that says which websites are allowed to talk to the backend.
Backend / serverCode that runs in the cloud, not in the browser. Handles secrets and data.
Frontend / clientThe HTML/JS the learner sees in their browser.
POST requestA message your frontend sends to the backend containing data (like a chat message).
Script: Requesting a New AI Endpoint

Use this when you want to add an AI-powered interaction to a course and need the dev team to set up the backend:

📋 What to say
"Hi [name], I'm building an AI-powered scenario for a course and I need a backend endpoint to keep the API key secure. Specifically, I need a POST endpoint — something like /chat — that accepts an array of messages in the standard OpenAI message format and returns the AI's response. The API key for [Grok/OpenAI/etc.] should be stored as an environment variable on the server, not in any code. I'll be calling it from a frontend hosted at [your URL], so CORS should be configured to allow requests from that origin only. Happy to share the frontend code so you can see exactly what format I'll be sending."
Script: Requesting a CORS Update

Use this when you've moved your course to a new platform and the AI calls are getting blocked:

📋 What to say
"The AI interaction in [course name] is no longer working because we moved the course to [new platform/URL]. The backend's CORS policy is blocking requests from the new origin. Could you add [new URL] to the allowed origins list in the server's CORS configuration? The current allowed origin is [old URL] — you can either replace it or add the new one alongside it if both need to work."
Script: Reporting a Broken Endpoint
📋 What to say
"The AI interaction in [course name] stopped responding as of [date/time]. Learners are seeing [describe the error or behavior]. The frontend is at [URL] and it's trying to reach the /chat endpoint at [backend URL]. I haven't changed anything on the frontend side. Could you check the server logs and confirm whether the issue is on the backend? I can also share browser console screenshots if that helps."
Script: Asking for a New API Key to Be Added
📋 What to say
"I have a new API key for [service] that needs to replace the existing one on the server. I'll send it to you securely — please store it as an environment variable named GROK_API_KEY (or whatever naming convention you use) and do not commit it to the repo. Once it's updated on Render/your hosting platform, the old key can be deactivated."
What to Include in Any Request

When you reach out to an engineer about a backend issue, always include:

  1. The frontend URL — where the course lives
  2. The backend URL — the endpoint being called
  3. What you expected to happen vs. what actually happened
  4. When it started — did it ever work, or is it new?
  5. Any error messages from the browser console (right-click → Inspect → Console tab)
Pro tip: Open your browser's DevTools (F12), go to the Network tab, and trigger the AI interaction. You'll see the actual request and response — including any error codes. Sharing a screenshot of this with an engineer will save everyone a lot of back-and-forth.
